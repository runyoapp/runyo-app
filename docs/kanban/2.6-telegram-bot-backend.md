# 2.6 — Telegram-bot reads from backend (i.p.v. Sheets)

**Fase:** 2 (Core features)
**Type:** AFK
**Lagen:** backend + bot
**Depends on:** 2.1
**Blocks:** —
**Status:** pending

## Doel

De `runyoapp/runyo-bot` repo loskoppelen van Google Sheets. Alle `gspread`-calls worden vervangen door HTTP-calls naar de runyo-backend, geauthenticeerd via `BOT_SECRET`. `/feedback`-flow in chat schrijft via `PATCH /api/activities/:id`. Bot blijft een Python-service op Railway; geen frontend-werk.

## Acceptance

- Bot stuurt notificaties met content uit backend-DB (zelfde tekst/format als nu, andere bron)
- `/feedback`-command in Telegram resulteert in een `feedback`+`rating` veld in de DB (controleerbaar via `GET /activities/:id`)
- Bot werkt voor users die nog niet Telegram gekoppeld hebben → skip silently zonder errors te loggen
- Backend-endpoint `GET /api/internal/users-with-telegram` (BOT_SECRET) retourneert lijst met `{ user_id, telegram_chat_id, schema_id, prefs }` zodat de bot in één call zijn werklijst krijgt
- `gspread`-dependency verwijderd uit `runyo-bot/requirements.txt`
- Tests: bot-unit tests op de backend-fetch-formatting (≥2 — happy + skip-on-no-telegram)
- Tests: backend integration test op `GET /api/internal/users-with-telegram` met en zonder `BOT_SECRET` (≥2)

## Test-strategie (red-green-refactor)

1. **Red** — schrijf `runyo-bot/tests/fetch_test.py` met een mocked HTTP-response (httpx-mock) voor `/api/internal/users-with-telegram` met 2 users (1 met telegram, 1 zonder); assert dat de bot's `build_notifications(...)` voor de no-telegram-user niets oplevert. Faalt want functie bestaat niet.
2. **Green** — implementeer in `runyo-bot/runyo_bot.py` (of equivalent) de nieuwe `BackendClient` class die alle Sheets-calls vervangt. Verwijder gspread. Implementeer in `runyo-auth` de twee nieuwe internal-endpoints (`GET users-with-telegram`, `PATCH activities/:id` met BOT_SECRET-auth-pad naast Google-token-pad).
3. **Refactor** — als de bot-side fetch en de app-side fetch in `services/activities.ts` parallelle shapes hebben, leg een gedeelde JSON-shape vast in de backend-response. Niet via een gedeelde codebase — gewoon contract-overeenkomst gedocumenteerd in een comment.

## Tech-hints

- `BOT_SECRET` is al een env-var in beide services (zie `CLAUDE.md` runyo-bot env-vars). Hergebruik die voor auth op de nieuwe internal-endpoints.
- Backend krijgt twee auth-paden op de activity-routes: Google-token (user) of BOT_SECRET (service). Een eenvoudige `requireAuthOrBot` middleware is genoeg.
- `users.telegram_username` + `users.telegram_chat_id` toevoegen aan `users` tabel (eerder al gepland via `/bot/register`-endpoint dat bestaat). Migratie hier.
- Valkuil: de huidige bot leest `BOT_SECRET` uit env; bij Railway-deploy moeten beide services dezelfde value hebben — niet vergeten te roteren als één wijzigt.
- Houd het oude `/user/settings` endpoint voor backwards compat tijdens overgang; verwijder pas nadat bot stabiel draait op de nieuwe endpoints (eigen mini-ticket later, niet hier).
