# 1.2 — Backend baseline + eerste schema-tracer

**Fase:** 1 (Foundation, sequentieel)
**Type:** AFK
**Lagen:** db + backend + frontend + visible
**Depends on:** 1.1
**Blocks:** 2.1, 2.3, 2.5
**Status:** completed

## Doel

Postgres + Drizzle ORM op Railway aanzetten in `runyo-auth`, eerste tabellen (`users`, `schemas`) en endpoints `POST /api/schemas` + `GET /api/schemas/me` opleveren. Frontend krijgt `services/schemas.ts` en `dataStore` switcht van Sheets-`sheetId` naar backend-`schemaId`. Zichtbaar resultaat: ingelogde user ziet "Nog geen schema" met een knop, tap maakt een leeg schema aan en de UI toont "Schema: 0 trainingen". Tijdelijke UI uitsluitend voor deze tracer — productie-flow gaat later via 2.2 (importer).

## Acceptance

- `POST /api/schemas` met geldige Google-token → 201 + `{ id }`
- `POST /api/schemas` zonder of met ongeldige token → 401
- `GET /api/schemas/me` → array met alleen schemas van de ingelogde user (multi-user-isolation aantoonbaar)
- Frontend toont na sign-in correct de empty-state ↔ schema-aanwezig-state op basis van `services/schemas.ts`
- `dataStore` gebruikt `schemaId` (uuid) i.p.v. `sheetId`; geen hardcoded Sheets-referenties meer in deze tracer-flow
- Tests: minimaal 3 backend integration tests — auth happy, auth missing/invalid, multi-user-isolation
- Tests: 1 unit test op `services/schemas.ts` met gemockte fetch (happy + 401 sad path)

## Test-strategie (red-green-refactor)

1. **Red** — schrijf een integration test in `runyo-auth/tests/schemas.test.ts` die `POST /api/schemas` aanroept zonder Authorization-header en `expect(res.status).toBe(401)` verwacht; daarna een test die met token van user-A een schema maakt en met token van user-B verifieert dat `GET /api/schemas/me` voor B een lege array geeft. Beide falen want endpoints bestaan nog niet.
2. **Green** — implementeer de Drizzle-schema's (`users`, `schemas`), de Google-token-verify middleware en de twee handlers. Test moet groen worden zonder de assert te versoepelen.
3. **Refactor** — extract token-verify naar een herbruikbare `requireAuth` middleware zodra een tweede endpoint er ook om vraagt; geen prematuur DRY-en in deze slice.

## Tech-hints

- Drizzle ORM gekozen i.p.v. Prisma (PRD §5.1) — lichter, query-builder-style past beter bij Express-handlers.
- Railway Postgres add-on toevoegen aan de bestaande `runyo-auth` Railway-service; `DATABASE_URL` env-var zet Drizzle automatisch goed.
- `users` tabel houdt nu al `google_sub`, `apple_sub` (nullable), `password_hash` (nullable) — zie PRD §5.2; 2.3 vult Apple/email pas in.
- Frontend te raken: `src/services/` (nieuw bestand), `src/state/dataStore.ts` (sheetId → schemaId), entry-screen krijgt tijdelijk de empty-state-knop. Verwijder de Sheets-tracer niet wereldwijd; alleen het pad dat door deze tracer loopt.
- Valkuil: Google-token-verify moet `aud` checken tegen de runyo OAuth client-id (zie `CLAUDE.md`). Een token van een andere Google-app mag niet werken.
