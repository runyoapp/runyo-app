## Doel

Vitest + npm-scripts + GitHub Actions CI opzetten in `runyo-app` zodat alle volgende tickets niet-cheatbare tests kunnen vereisen. Eerste test op een bestaande pure service (`src/utils/date.ts`) als bewijs dat de runner werkt.

## Acceptance

- `npm run typecheck` werkt en is groen
- `npm run test` draait met ≥1 passerende test op een functie uit `src/utils/date.ts`
- `npm run lint` werkt (eslint, `--max-warnings 0`)
- `npm run check` (typecheck && test && lint) is groen lokaal én op CI
- CI workflow (`.github/workflows/check.yml`) draait op elke push naar `rn-migration` en `main`
- Een kunstmatig falende test zorgt dat CI rood wordt — sanity-bewijs in PR-beschrijving (screenshot of CI-link)

## Test-strategie (red-green-refactor)

1. **Red** — schrijf een test in `src/utils/date.test.ts` die een bestaande functie aanroept op een case die fout terugkomt (of forceer met `expect(...).toBe(...)` op een verkeerde verwachting).
2. **Green** — corrigeer ofwel de test naar de juiste verwachting (als functie correct is) ofwel de functie (als de test een echte bug ontmaskert).
3. **Refactor** — extract gedeelde test-helpers alleen als de test-file daadwerkelijk meerdere herhalingen krijgt; geen prematuur DRY-en.

## Tech-hints

- Vitest gekozen i.p.v. Jest (PRD §6.1) — Expo's Jest-preset is zwaar en RN-specifiek; Vitest is sneller en past op pure TS-services.
- `vitest.config.ts` met `environment: 'node'` voor service-tests; RN-component-tests komen later in fase 2 met React Native Testing Library (apart config-block).
- Scripts in `package.json`: `test`, `test:watch`, `typecheck`, `lint`, `format`, `check`.
- GitHub Actions: Node 20, cache `~/.npm` op `package-lock.json` hash, run `npm ci && npm run check`.
- Tests die de agent niet zelf kan editen vlak voor commit: maak `npm run check` de enige geaccepteerde groen-poort, en zet test-bestanden niet in een `.gitattributes merge=ours` of vergelijkbare omleiding.
