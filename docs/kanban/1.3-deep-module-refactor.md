# 1.3 — Deep-module refactor (drie shallow screens splitsen)

**Fase:** 1 (Foundation, parallel met 1.2)
**Type:** AFK
**Lagen:** frontend only (refactor, geen nieuwe features)
**Depends on:** 1.1
**Blocks:** 2.2, 2.4
**Status:** pending

## Doel

De drie te-shallow screens uit PRD §5.4 splitsen naar deep modules met simpele interfaces. Geen functionele wijziging — de bestaande flows blijven zichtbaar identiek. Maakt 2.2 (importer-naar-backend) en 2.4 (edits-flow) leesbaarder en testbaar door de nieuwe service-files te vervangen i.p.v. UI-blobs te bewerken.

## Acceptance

- `ImportModal` (huidig 457r) gesplitst in `src/services/import.ts` (pure parse-functies per format) + `src/screens/ImportFlow.tsx` (step-orchestratie + rendering)
- `TodayScreen` (huidig 234r) gesplitst in `src/hooks/useTodayData.ts` (afleidingen) + `src/hooks/useDaySwipe.ts` (PanResponder) + `src/screens/TodayScreen.tsx` (compositie)
- `DayDetailModal` (huidig 264r) gesplitst in `src/services/activityEdit.ts` (validatie + API-calls) + `src/screens/DayDetailModal.tsx` (UI-shell)
- Bestaande functionaliteit ongewijzigd — manuele QA op alle drie flows (import-happy-path, swipe vooruit/terug op Today, edit-en-save in DayDetail)
- Service-files zijn pure functies: geen `react-native`/UI-imports, geen `useState`/`useEffect`
- Tests: per nieuwe service minimaal 2 unit tests (happy + 1 randgeval)
- Geen file in `src/screens/` > 200 regels na refactor (richt-getal, niet hard)
- `npm run check` blijft groen

## Test-strategie (red-green-refactor)

1. **Red** — schrijf `src/services/import.test.ts` met een test die `parsePdf(mockText)` aanroept en een array met >0 activities verwacht; analoog voor `activityEdit.test.ts` (`validateActivity({ datum: 'invalid' })` → ValidationError). Beide falen want functies bestaan nog niet als pure export.
2. **Green** — extract de logica uit de huidige modal-files naar de nieuwe service/hook-files; importeer ze terug in de screens; runtime-gedrag mag niet veranderen. Tests groen.
3. **Refactor** — als er duplicatie tussen parse-functies blijkt (bv. een common `normalizeRow`), extract die naar `src/services/import.utils.ts`. Geen prematuur DRY-en.

## Tech-hints

- Werk pure-eerst: trek de logica los, run de tests, koppel daarna pas de UI weer aan. Voorkomt dat een UI-bug zich vermomt als een service-bug.
- `useTodayData` en `useDaySwipe` mogen `useMemo`/`useState` gebruiken — het zijn hooks, niet services. Verschil: hooks zijn React-gebonden, services zijn dat niet.
- `activityEdit.ts` doet straks ook `PATCH`-calls naar backend (2.1) — houdt de API-call apart van validatie zodat 2.1 alleen de network-laag hoeft te swappen.
- Valkuil: PanResponder-state in `useDaySwipe` moet niet via `useRef` naar buiten lekken naar `TodayScreen`. Hook returnt alleen de gesture-handlers + de huidige dag-index.
- Geen routes/navigatie herstructureren — buiten scope.
